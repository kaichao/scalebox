## 6.3 本地计算（Local Compute）

- task_dist_mode: 
  - HOST-BOUND: headers->>'to_host'
  - SLOT-BOUND
  - GROUP-BOUND

硬件条件：
- 需要真正的本地存储支持，云上的虚机不适合本地计算。
- 自建服务器集群（不带虚拟化）
- HPC计算节点


本地计算的主要技术问题
- 前后module间的流控
  - 运行速度匹配，避免本地硬盘/内存爆满
  - 信号量是实现流控的技术手段
- 同一module在多节点间同步流控
  - 不同通道的处理/不同节点的数据下载
  - 节点处理能力差异，因此中间数据累积，导致内存爆满
- 自动容错处理
  - 出错影响部分数据的处理
  - 流控会导致整个流水线停止
  - 自动容错对流控的影响
- 关键module的task处理排序
- 非关键模块分解
  - 细分会增加流控负担
  - 增加流水线并行提升性能，会增加中间存储的内存使用，影响流水线运行的稳定性

## 6.6 高级流控管理

对于复杂应用程序的本地计算，高级流控可大大简化流控逻辑，避免运行异常。

### 6.6.1 分组的node间并行同步流控
- 标准流控属性```task_progress_diff```
  - 按节点的进度计数器，基于同步信号量实现
  - task_progress_diff：当前节点进度与最慢进度的差值作为控制变量

- 消息路由中，按task处理进度，修改对应信号量；
- 业务模块基于信号量计数值，实现同步流控；

### 6.6.2 vtask计数流控

#### vtask定义

vtask标识一种跨越多个模块的虚拟task，是基于计数的流控机制，分为全局计数（global_vtask）、分组计数（group_vtask）、节点计数（host_vtask）等三类。

启用vtask的模块，在task启动时将计数器自动减一，若计数器为0，则计数流控生效，新task不能运行。从实现上看，vtask机制设置了等待队列长度。

后续模块的消息处理完成后，在消息路由中自动恢复vtask计数器。也可对计数器手工操作，实现启停。

- 处理过程
  - 在模块入口处，检查信号量计数器，若小于等于0，表示无运行槽位，流控不通过；
  - 该给后续消息路由发送消息中，加上消息头```global_vtask / group_vtask / host_vtask```
  - 消息路由对该计数值自动减一，占用一个运行槽位。该操作自动完成，不需要编程控制；
  - vtask处理完成，则在对应消息路由中，通过信号量操作，对计数值加一，释放一个空槽位。

#### global_vtask
- 属性名：```global_vtask_size```，后续模块的等待队列长度。

- yaml应用文件解析时，生成对应信号量：```global-vtask-size_${mod_name}```，其初值为参数值。
  - mod_name为首模块名，模块通常运行在头节点上，仅有1个全局slot。

#### 按计算节点组的vtask数量流控
- 属性名：```group_vtask_size```，计算节点组上最多可运行vtask数量，表示最多可进入运行的vtask数量。

- yaml应用文件解析时，生成对应信号量：```group-vtasks-size_${mod_name}_${groupname}```，其初值为参数值。
  - mod_name为首模块名，首模块通常为头节点上运行的模块，全系统中仅有1个对应的slot
  - groupname为t_host表中对应的gid

#### 按计算节点的vtask数量流控
- 属性名：```host_vtask_size```，计算节点上最多可运行的vtask数量，表示节点上运行队列的长度。

- yaml应用文件解析时，生成对应信号量：```host-vtask-size_${mod_name}_${hostname}```，其初值为参数值。
  - mod_name为首模块名，首模块通常为HOST-BOUND，每节点上仅有1个slot
  - hostname为slot对应的hostname

## 6.8 容错机制

### 6.8.1 模块级容错
- 设置module的参数retry_rules。

### 6.8.2 平台级故障及恢复

- slot启动运行出错，但运行task时很快出错，导致task运行时间短，单位时间内可完成多个task；
- module的max_tasks_per_minutes参数（取值>=3），在超出该参数时，将对应slot设置为'ERROR'，以免干扰其他task的正常等待；
- 再结合retry_rules，即可在有大量slot运行错的情况下，停止有问题的slot运行；
- 最后，可以再通过手工恢复。

## 6.9 slot自动扩缩容

### 6.9.1 概述

Slot 自动扩缩容是 ScaleBox 平台的核心特性之一，用于根据资源使用情况自动调整任务执行槽（slot）的数量。系统通过监控主机资源使用率，智能决策何时扩容（增加slot）或缩容（减少slot），以优化资源利用率和任务执行效率。

### 6.9.2 功能特性

#### 核心功能
- **自动监控**：定期检查主机资源使用情况
- **智能决策**：基于使用率阈值自动决定扩容/缩容
- **双重策略**：支持资源紧张扩容和容量不足扩容
- **安全保护**：防止过度扩容和过度缩容
- **冷却机制**：避免频繁调整，确保系统稳定

#### 扩缩容场景
1. **资源紧张扩容**：当资源使用率超过阈值时扩容
2. **容量不足扩容**：当slot数量远低于最大容量时扩容
3. **资源闲置缩容**：当资源使用率过低时缩容
4. **超量运行缩容**：当slot数量超过物理容量时强制缩容

### 6.9.3 实现原理

#### 系统架构
```
+----------------+     +----------------+     +----------------+
|   资源监控模块   | --> |   决策算法模块   | --> |   Slot调整模块  |
+----------------+     +----------------+     +----------------+
        |                       |                       |
        v                       v                       v
+----------------+     +----------------+     +----------------+
| 收集主机资源数据 |     | 计算扩缩容决策  |     |  执行slot调整   |
+----------------+     +----------------+     +----------------+
```

#### 数据流
1. **资源收集**：从数据库获取主机总资源、已使用资源和slot需求
2. **指标计算**：计算最大可用slot数量和资源使用率
3. **决策判断**：基于阈值判断是否需要调整
4. **执行调整**：通过slot管理接口调整slot数量

### 6.9.4 算法说明

#### 核心算法流程

```go
// 五步算法流程
1. 计算最大可用slot数量
2. 计算最大资源使用率
3. 检查是否需要扩容
4. 检查是否需要缩容
5. 返回调整决策
```

#### 详细算法步骤

##### 1. 计算最大可用slot数量
- 获取总资源、已使用资源、slot需求的交集
- 对每个共同指标：`可用slot = (总资源 - 已使用资源) / slot需求`
- 取所有指标中的最小值作为最大可用slot数量
- 应用配置限制（最小/最大slot数量）

##### 2. 计算最大资源使用率
- 对每个共同指标：`使用率 = 已使用资源 / 总资源`
- 取所有指标中的最大值作为决策依据

##### 3. 扩容检查（双重条件）
- **条件A：资源紧张扩容**
  - 触发：`使用率 > 扩容阈值`
  - 目标：缓解资源压力
- **条件B：容量不足扩容**
  - 触发：`当前slot数量 < 最大可用slot数量 × 容量阈值`
  - 目标：充分利用可用容量

##### 4. 缩容检查（双重条件）
- **条件A：资源闲置缩容**
  - 触发：`使用率 < 缩容阈值`
  - 目标：减少资源浪费
- **条件B：超量运行缩容**
  - 触发：`当前slot数量 > 最大可用slot数量`
  - 目标：强制缩容到物理容量内

##### 5. 调整数量计算
- **扩容数量**：`min(最大可用 - 当前, 扩容步长)`
- **缩容数量**：`min(当前 - 目标, 缩容步长)`
  - 资源闲置缩容：目标 = 最小slot数量
  - 超量运行缩容：目标 = 最大可用slot数量

### 6.9.5 配置说明

#### 启用配置
在模块参数中设置以下配置项：

```json
{
  "autoscaling.enabled": "true",
  "autoscaling.min_slots": "1",
  "autoscaling.max_slots": "10",
  "autoscaling.scale_up_threshold": "0.8",
  "autoscaling.scale_down_threshold": "0.3",
  "autoscaling.scale_up_step": "2",
  "autoscaling.scale_down_step": "1",
  "autoscaling.scale_up_cooldown_seconds": "300",
  "autoscaling.scale_down_cooldown_seconds": "600",
  "autoscaling.check_interval_seconds": "60",
  "autoscaling.capacity_threshold": "0.8"
}
```

#### 参数详解

##### 基础参数
| 参数名 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `autoscaling.enabled` | boolean | false | 是否启用自动扩缩容 |
| `autoscaling.min_slots` | int | 1 | 最小slot数量（缩容下限） |
| `autoscaling.max_slots` | int | 10 | 最大slot数量（扩容上限） |

##### 阈值参数
| 参数名 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `autoscaling.scale_up_threshold` | float | 0.8 | 扩容阈值：使用率 > 此值时扩容 |
| `autoscaling.scale_down_threshold` | float | 0.3 | 缩容阈值：使用率 < 此值时缩容 |
| `autoscaling.capacity_threshold` | float | 0.8 | 容量阈值：当前slot < 最大可用 × 此值时扩容 |

##### 步长参数
| 参数名 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `autoscaling.scale_up_step` | int | 2 | 每次最多扩容的slot数量 |
| `autoscaling.scale_down_step` | int | 1 | 每次最多缩容的slot数量 |

##### 时间参数
| 参数名 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `autoscaling.scale_up_cooldown_seconds` | int | 300 | 扩容冷却时间（秒） |
| `autoscaling.scale_down_cooldown_seconds` | int | 600 | 缩容冷却时间（秒） |
| `autoscaling.check_interval_seconds` | int | 60 | 检查间隔时间（秒） |

### 6.9.6 资源指标要求

#### 数据格式
系统需要以下三类资源指标：

##### 1. 主机总资源（total.*）
```json
{
  "total.compute_units": "100",
  "total.memory_mb": "10240",
  "total.disk_gb": "500"
}
```
该项配置位于主机配置的parameters中。

##### 2. 主机已使用资源（used.*）
```json
{
  "used.compute_units": "50",
  "used.memory_mb": "5120",
  "used.disk_gb": "250"
}
```
该项配置位于主机配置的metrics中，由外部监控系统定时更新。

##### 3. Slot资源需求（slot_req.*）
```json
{
  "slot_req.compute_units": "1",
  "slot_req.memory_mb": "256",
  "slot_req.disk_gb": "10"
}
```
该项配置位于模块配置的parameters中。

#### 指标要求
1. **指标一致性**：三类指标需要有相同的key前缀（去掉前缀后）
2. **数值类型**：所有值必须是数值类型（整数或浮点数）
3. **单位一致**：相同指标的单位必须一致

### 6.9.7 使用示例

#### 场景1：正常扩容
```
初始状态：当前slot=3，最大可用slot=10，使用率=90%
配置：扩容阈值=80%，容量阈值=80%

决策过程：
1. 使用率90% > 80% → 触发资源紧张扩容
2. 当前3 < 10×80%=8 → 触发容量不足扩容
3. 可扩容数量：min(10-3=7, 扩容步长=2) = 2
结果：扩容2个slot
```

#### 场景2：正常缩容
```
初始状态：当前slot=8，最大可用slot=10，使用率=20%
配置：缩容阈值=30%

决策过程：
1. 使用率20% < 30% → 触发资源闲置缩容
2. 可缩容数量：min(8-1=7, 缩容步长=1) = 1
结果：缩容1个slot
```

#### 场景3：超量运行缩容
```
初始状态：当前slot=5，最大可用slot=1，使用率=88%
配置：缩容阈值=30%

决策过程：
1. 当前5 > 最大1 → 触发超量运行缩容（无论使用率）
2. 可缩容数量：min(5-1=4, 缩容步长=1) = 1
结果：缩容1个slot
```

### 6.9.8 监控指标
建议监控以下关键指标：
1. **资源使用率**：各资源的使用率变化
2. **slot数量**：当前slot数量的变化趋势
3. **扩缩容频率**：扩容/缩容操作的频率
4. **决策延迟**：从检测到调整的时间延迟

### 6.9.9 最佳实践

#### 配置建议
1. **阈值设置**：
   - 扩容阈值：70%-90%（根据业务敏感性）
   - 缩容阈值：20%-40%（保留适当缓冲）
   - 容量阈值：70%-90%（避免过早扩容）

2. **步长设置**：
   - 扩容步长：2-5（避免一次性扩容过多）
   - 缩容步长：1-2（谨慎缩容，避免影响业务）

3. **冷却时间**：
   - 扩容冷却：300-600秒（5-10分钟）
   - 缩容冷却：600-1200秒（10-20分钟，更长以保持稳定）

#### 资源规划
1. **指标选择**：选择最能反映业务压力的资源指标
2. **需求估算**：准确估算每个slot的资源需求
3. **容量预留**：保留20%-30%的缓冲容量应对突发负载

#### 故障处理
1. **资源不足**：系统检测到资源不足时会停止扩容
2. **指标异常**：指标缺失或异常时会跳过该主机
3. **配置错误**：配置解析失败时会使用默认值

### 6.9.10 限制与注意事项

#### 系统限制
1. **指标依赖**：需要完整的三类资源指标
2. **精度限制**：slot数量为整数，可能无法完全匹配资源
3. **延迟性**：从资源变化到调整完成有检查间隔+冷却时间

#### 使用注意事项
1. **避免频繁调整**：合理设置冷却时间和检查间隔
2. **监控告警**：设置关键指标的告警阈值
3. **容量规划**：确保物理资源足够支持最大slot数量
4. **测试验证**：在生产环境前充分测试扩缩容逻辑
